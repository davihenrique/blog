## Introdução

**CRLF** e **LF** são caracteres utilizados para sinalizar uma quebra de linha em arquivos de texto puro. A convenção varia conforme o sistema operacional:

- **Windows:** `CRLF` (`\r\n`)
- **Linux / Unix / macOS (atual):** `LF` (`\n`)
- **macOS (antes de ~2001):** `CR` (`\r`)

A origem dessa diferença remonta aos tempos das máquinas de escrever. Ao chegar ao fim de uma linha, eram necessários dois movimentos distintos:

- **Carriage Return (CR):** Retornar o "carro" ao início da linha.
- **Line Feed (LF):** Avançar o papel para a próxima linha.

O Windows herdou essa lógica e manteve os dois caracteres juntos (`\r\n`). Os sistemas Unix, por outro lado, simplificaram o padrão e passaram a usar apenas o `\n`.

---

## O Problema na Prática

Certa vez, clonei um repositório do GitHub e fui executar um container com o Docker. Ocorreu um erro que, a princípio, não fazia sentido. Ao investigar, percebi que o projeto utilizava scripts `.sh` em algumas etapas e esses arquivos estavam com quebras de linha no formato **CRLF**.

O problema é que o shell do Linux não reconhece o `\r` como parte do comando, ele o interpreta como um caractere inválido, o que quebra a execução do script. Simplesmente converter os arquivos para **LF** já foi suficiente para tudo funcionar.

O que me pegou de surpresa foi descobrir que o Git no Windows tem por comportamento padrão ao clonar um repositório, ele converte automaticamente os arquivos de **LF** para **CRLF**. E ao fazer o push, converte de volta para **LF**. Na maior parte do tempo isso é transparente, porém as vezes pode acontecer problemas.

## Git e as Quebras de Linha

### O Problema

O conflito surge quando você desenvolve no Windows e faz o deploy em um servidor Linux. Se um arquivo subir com `CRLF`, scripts `.sh` podem falhar, e algumas ferramentas Linux simplesmente não lidam bem com o `\r` extra no final de cada linha.

---

### Como Resolver

#### 1. Conversão automática (recomendado para Windows)

Deixar o Git converter os arquivos automaticamente é a abordagem mais prática para quem desenvolve no Windows e faz deploy no Linux:
```bash
git config --global core.autocrlf true
```

Com essa configuração, o Git converte `LF → CRLF` ao fazer checkout e `CRLF → LF` ao fazer commit. 
O repositório remoto fica sempre com `LF`, e sua máquina Windows trabalha com `CRLF`.

---

#### 2. Desativar a conversão automática

Se você prefere que o Git não toque nas quebras de linha — útil quando você já controla isso manualmente ou usa ferramentas que fazem isso por você:
```bash
git config --global core.autocrlf false
```

Nesse caso, os arquivos são commitados exatamente como estão no seu sistema. 
Use com cautela: é você quem garante que o formato está correto antes de commitar.

---

#### 3. Forçar LF em todos os arquivos

Para converter tudo para `LF` de uma vez e garantir que o repositório fique consistente:
```bash
git config --global core.autocrlf input
```

Com `input`, o Git converte `CRLF → LF` ao fazer commit, mas não faz conversão alguma no checkout.

---

#### 4. Usar um arquivo `.gitattributes`

A solução mais robusta é definir o comportamento diretamente no repositório, independentemente da configuração local de cada desenvolvedor.
Crie um arquivo `.gitattributes` na raiz do projeto:

```
# Normaliza tudo para LF no repositório
* text=auto eol=lf

# Scripts shell sempre com LF
*.sh text eol=lf

# Arquivos Windows que precisam de CRLF
*.bat text eol=crlf
*.cmd text eol=crlf

# Binários: sem conversão
*.png binary
*.jpg binary
*.gif binary
*.ico binary
*.zip binary
*.exe binary
```

A diretiva `text=auto eol=lf` instrui o Git a detectar automaticamente se um arquivo é texto, em caso positivo, usar `LF` como padrão.
Com isso, todos no projeto — independentemente do sistema operacional — commitem sempre com `LF`.
